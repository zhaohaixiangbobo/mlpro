# 优化预测功能：文件名规范化与状态持久化

针对您的两个需求，我将制定以下实施计划：

## 1. 优化预测结果文件名 (Backend)

**目标**: 生成具有辨识度且长度可控的文件名，格式为 `{原始文件名}_{模型名}_{时间戳}.{ext}`。

**修改方案 (`backend/app/services/ml_service.py`)**:
*   修改 `predict` 方法中的文件名生成逻辑。
*   **新格式**: `pred_{original_name_short}_{model_type}_{timestamp}.{ext}`
    *   `original_name_short`: 截取原始文件名的前10个字符（去扩展名），防止过长。
    *   `model_type`: 从模型对象中获取类名（如 `RandomForestClassifier` -> `RF`）或由前端传递简写。为简化实现，我们将从前端传递算法名称简写（如 `RF`, `LR`）。
    *   `timestamp`: 使用 `YYYYMMDD_HHMM` 格式，精确到分，易读且唯一。

**API 变更 (`backend/app/api/workflow.py`)**:
*   `PredictRequest` 模型增加 `algorithm_label` 字段（可选），用于生成文件名。

## 2. 预测状态持久化 (Frontend)

**目标**: 刷新页面或重新进入时，恢复上一次的预测配置（工作流、模型、文件）和预测结果。

**修改方案 (`frontend/src/pages/Prediction.tsx`)**:
*   **使用 `localStorage`**:
    *   在状态变更时（`handleWorkflowChange`, `setSelectedAlgoId`, `setSelectedFile`, `setPredictionResult`），将这些状态保存到 `localStorage`。
    *   键名示例: `mlpro_prediction_state`。
*   **初始化恢复**:
    *   在 `useEffect` 初始化时，读取 `localStorage`。
    *   如果存在缓存状态，自动填充 `selectedWorkflow`, `selectedAlgoId`, `selectedFile` 和 `predictionResult`。
    *   **注意**: 恢复工作流选择后，需要自动触发 `handleWorkflowChange` 中的逻辑（加载工作流详情并解析算法节点），以确保 `algorithms` 列表被正确填充，否则下拉框只显示 ID 没有选项。

## 3. 执行步骤

1.  **Backend**: 修改 `ml_service.py` 的文件名生成逻辑，使用时间戳和模型标识。
2.  **Backend**: 更新 `workflow.py` 的 `PredictRequest` 定义，接收算法标签。
3.  **Frontend**: 更新 `api.ts` 的 `predict` 方法，传递 `algorithm_label`。
4.  **Frontend**: 修改 `Prediction.tsx`：
    *   实现 `saveState` 和 `loadState` 逻辑。
    *   在 `useEffect` 中处理状态恢复，特别是需要重新加载工作流数据以恢复 `algorithms` 列表。

通过以上修改，预测文件的命名将更加规范易读，且用户的操作上下文将得到保留，提升使用体验。
