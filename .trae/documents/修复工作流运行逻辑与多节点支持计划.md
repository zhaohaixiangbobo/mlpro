# 修复工作流运行逻辑与多节点支持计划

## 问题分析
1.  **后端报错 (Unknown algorithm: 缺失值处理)**:
    *   **原因**: 前端 `handleRun` 函数使用 `nodes.find(n => n.type === 'algoNode')` 查找算法节点。由于“缺失值处理”节点的类型也是 `algoNode` (在 Sidebar 中定义)，如果它被先找到，前端就会将其作为主算法发送给后端。后端 `run_algorithm` 无法识别“缺失值处理”这个算法名，从而抛出异常。
    *   **解决**: 在查找算法节点时，必须过滤 `category === 'Model'`，排除预处理节点。

2.  **多条线/多终端节点支持**:
    *   **原因**: 目前代码只查找**一个**算法节点并运行，无法支持多个分支或并发运行。
    *   **解决**: 修改逻辑为查找所有 `category === 'Model'` 的节点，针对每一个模型节点单独回溯路径并执行运行请求。

3.  **预处理参数丢失**:
    *   **原因**: 目前的 `handleRun` 没有收集路径上的预处理节点信息。
    *   **解决**: 实现回溯算法 (Backtracking)，从模型节点向上寻找数据节点，沿途收集预处理节点并构建 `preprocessing` 参数列表。

## 实施计划

### 1. 修改 `Workflow.tsx` 中的 `handleRun`
重构 `handleRun` 函数，使其支持：
*   **识别所有终端模型**: `const modelNodes = nodes.filter(n => n.type === 'algoNode' && n.data.category === 'Model');`
*   **路径回溯**: 为每个模型节点实现 `traceBackPath(node)` 函数，查找其对应的数据源和中间的预处理步骤。
*   **并发执行**: 使用 `Promise.all` 并发执行所有模型节点的运行请求。
*   **预处理映射**: 将预处理节点的参数转换为后端 `ml_service.py` 期望的格式 (提取 `method` 字段)。

### 2. 实现辅助函数 `traceBackPath`
*   输入: `startNode` (模型节点)
*   输出: `{ dataNode, preprocessingNodes }`
*   逻辑:
    *   利用 `edges` 查找输入源。
    *   如果是 `algoNode` 且 `category === 'Preprocessing'`，加入预处理列表。
    *   如果是 `dataNode`，确认为数据源。

### 3. 验证
*   **测试场景 1**: 单一流程 (数据 -> 缺失值处理 -> 逻辑回归)。确保不再报错，且预处理生效。
*   **测试场景 2**: 多分支流程 (数据 -> 逻辑回归 / 数据 -> 决策树)。确保两个算法都运行并产生结果。
*   **测试场景 3**: 错误处理。如果路径断开或缺少数据节点，给予提示。

该计划将直接解决您遇到的 500 错误，并实现您期望的多条线并发运行功能。
