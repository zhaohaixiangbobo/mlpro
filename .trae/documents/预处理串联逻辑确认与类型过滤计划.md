# 数据预处理逻辑确认与类型过滤优化计划

针对您的两个问题，我的分析结论和计划如下：

## 1. 预处理节点串联逻辑确认

*   **结论**: **是的，实际运行时第二个节点是在第一个节点处理结果的基础上进行操作的。**
*   **依据**: 后端服务 (`ml_service.py`) 采用顺序执行机制。它接收一个预处理步骤列表，并依次对 DataFrame 进行原地修改。因此，如果您串联了 [缺失值处理 -> 标准化]，数据流向是：原始数据 -> 缺失值处理 -> 中间数据 -> 标准化 -> 最终数据。

## 2. 基于列类型的智能过滤优化计划

为了解决“标准化只能处理数值型”以及“均值填充不适合类别型”的问题，我将对前端参数面板进行以下改造：

### 核心改动：前端获取列类型信息

*   目前 `PropertyPanel` 仅获取了列名。后端 API (`/data/preview`) 实际上在 `meta` 字段中返回了详细的列统计信息（包含 `type`: "numeric" | "category"）。
*   我们将修改 `PropertyPanel` 以解析并向下传递完整的列信息对象 `{ title: string, type: string }`。

### 具体组件优化

1.  **标准化配置 (`StandardizationForm.tsx`)**:
    *   **逻辑**: 强制过滤。
    *   **实现**: 仅显示 `type === 'numeric'` 的列供选择。非数值型列将不会出现在选项中，防止用户误选导致报错。

2.  **缺失值处理配置 (`MissingValueForm.tsx`)**:
    *   **逻辑**: 动态过滤。
    *   **实现**: 
        *   当选择 **均值填充 (Mean)** 或 **中位数填充 (Median)** 时，仅显示数值型列。
        *   当选择 **众数填充 (Mode)** 或 **删除行 (Drop)** 时，显示所有列。
    *   添加交互：当用户切换处理方式时，自动更新可用的列列表。

## 执行步骤

1.  **修改 `PropertyPanel.tsx`**: 
    *   定义 `ColumnInfo` 接口。
    *   从 API 响应的 `meta.columns` 中提取列类型信息。
    *   将包含类型的列列表传递给子组件。
2.  **修改 `StandardizationForm.tsx`**: 
    *   接收带类型的列列表。
    *   渲染时过滤出数值列。
3.  **修改 `MissingValueForm.tsx`**: 
    *   根据当前选择的 `method` 动态过滤展示的列。

这将从根本上确保配置的合法性，提升用户体验。
